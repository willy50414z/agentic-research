## 審查步驟

依照上方的審查規則，整合 user 回覆並產出最終版本：

1. **整合留言回覆**
   從上方 Q&A 紀錄中提取 user 的回答，將答案填入 spec 對應欄位。
   若 user 的回答對某問題仍不足夠，保留該問題為待釐清。

2. **逐一核對必填欄位**
   以上方範例 spec 為格式基準，確認以下欄位均已完整且精確
   （不得出現「適當」、「最佳化」、「TBD」、「視情況而定」）：
   - 市場論點
   - 交易範圍（標的、資產類別、交易所、方向、時間週期、交易時段）
   - 資料規格（來源、必要欄位、訓練／驗證／測試期間、清理規則）
   - 進場訊號（精確條件：指標數值、交叉、閾值）
   - 出場訊號（精確條件 + 停損）
   - 倉位 Sizing 與風險規則
   - 執行假設（訂單類型、滑價、手續費、延遲、再平衡頻率）
   - 績效門檻（min win rate、max drawdown、min alpha ratio、profit factor IS/OOS）
   - Plugin（固定為 `quant_alpha`）

   仍缺漏但可合理推斷的欄位，填入並加註 *(假設)*，在 `## 假設說明` 條列原因。

3. **檢查回測標準符合性**
   確認進出場訊號、績效門檻、執行假設是否精確且可量化。
   若仍有欄位模糊、矛盾、或缺少無法合理推斷的關鍵商業決策，列為待釐清問題。

4. **依照範例 spec 的格式撰寫最終 `reviewed_spec_final.md`**（不含任何 Q&A 對話內容）

## 輸出

用工具將以下檔案寫入 `{OUTPUT_DIR}`：

- 必寫：`{OUTPUT_DIR}/reviewed_spec_final.md`
- 二擇一寫（不可同時）：
  - 所有問題已解決，spec 完整 → `{OUTPUT_DIR}/status_pass.txt`（內容：PASS）
  - 仍有待釐清問題 → `{OUTPUT_DIR}/status_need_update.txt`（每行一個問題）

## 補充校正要求

5. **逐題核對，不得重問已回答問題**
   先將上一輪每一個待釐清問題與 user 回覆逐一對照，標記為：
   - 已解決：答案已足以落地到 spec
   - 未解決：答案不足、互相矛盾、或仍缺關鍵決策
   `status_need_update.txt` 只能保留未解決問題，不可重複列出已解決項目。

6. **以最新 user 回答覆蓋舊假設**
   若 user 回答推翻了 Author 輪的推斷值，必須更新 spec 內容，並同步改寫 `## 假設說明`，
   移除失效假設，避免 final spec 同時保留新答案與舊假設而產生矛盾。

7. **Refine 輪不重做整份 spec 發散式審查**
   你的重點是確認 user 回答是否真正解掉前一輪問題，並讓 final spec 可直接進入 Verify。
   除非發現 user 回答導致新的內部矛盾，否則不要新增與本輪 Q&A 無關的新問題。
