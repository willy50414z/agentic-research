# 策略知識架構設計討論

_整理自 2026-04-02 設計討論_

---

## 核心目標

讓每次策略研究不只產出回測結果，而是累積可重用的知識。
知識以結構化方式儲存，讓後續策略可以查詢、借鑑、避免重複踩坑。

---

## 一、整體知識架構

### 三層結構

```
Layer 1：單策略知識
  strategies/<slug>/
    ├── spec.md                  ← 假說
    ├── capability-profile.md   ← 策略能力檔案（含因子組成、分數、痛點）
    ├── reports/                 ← 回測報告
    └── experiments/             ← 實驗紀錄

Layer 2：跨策略登錄表（共同查詢路徑）
  knowledge-base/knowledge/strategy-registry/
    ├── index.md                 ← 所有策略能力索引
    ├── factor-library.md        ← 因子庫（含環境 IC 評分）
    └── pain-point-routing.md   ← 痛點 → 解法路由表

Layer 3：已萃取教訓（持久化）
  knowledge-base/knowledge/backtest-records/
    └── <topic>.md              ← 跨策略觀察（現有結構不變）
```

### 設計原則

- **知識不存在 context，存在檔案**：重開 session 仍可查詢
- **痛點路由是因子庫的快取層**：第一次診斷後存起來，之後直接查
- **分數來自回測數據，非主觀填寫**：可重算、會自動更新

## 二、因子庫（Factor Library）

### 概念

將策略分解為最小粒度的因子，每個因子單獨評分，再依市場狀態加權組合。

**因子 ≠ 指標**。一個指標可能包含多個因子語義，需要拆開：

```
均線  →  拆成兩個純因子
  因子 1：趨勢方向  = 均線斜率 / 多空排列（輸出 [-1, 1]）
  因子 2：趨勢強度  = 價格偏離均線幅度（輸出 [0, 1]）
```

### 建議起步的 5 個因子

| 因子 | 衡量什麼 | 輸出型態 | 範例實作 |
|------|---------|---------|---------|
| 趨勢方向 | 均線斜率 / 排列 | 方向 [-1, 1] | EMA(20) 斜率正負 |
| 趨勢強度 | 趨勢的力道 | 強度 [0, 1] | ADX 正規化至 0–1 |
| 動能 | 近期價格加速度 | 方向 + 強度 | ROC(10) |
| 波動率狀態 | 市場是趨勢還是盤整 | 環境標籤 | BB Width 分位數 |
| 成交量確認 | 量是否配合價 | 確認訊號 [0, 1] | 量比（當日量 / 20日均量）|

### 因子評分方式

**不用主觀填分，改用 IC（資訊係數）**：

```
因子 IC = 因子訊號與未來 N 根 K 棒報酬的 Spearman 相關係數
```

對每個因子分別計算不同市場環境下的 IC：

| 因子 | 趨勢市場 IC | 盤整市場 IC | 高波動 IC | 低波動 IC |
|------|-----------|-----------|---------|---------|
| 趨勢方向 | 0.18 | 0.03 | 0.12 | 0.15 |
| 波動率狀態 | 0.09 | 0.21 | 0.07 | 0.19 |
| … | … | … | … | … |

組合時，依當前市場狀態選對應欄的 IC 做加權，IC 低的因子權重自動降低。

### 待決定的設計問題

**Q1：因子的粒度要怎麼定？**

- 選項 A：固定拆到「單一語義」（最細），例如方向、強度分開
- 選項 B：讓回測結果決定，如果「方向 + 強度」合在一起 IC 更穩，就不拆
- 選項 C：兩層並存，細粒度因子組成粗粒度因子模組

> 範例答案：建議先走選項 A（最細粒度），拆開後如果發現兩個因子 IC 高度相關才合併，而不是一開始就合在一起。

**Q2：IC 的預測窗口 N 怎麼選？**

- 選項 A：固定 N（例如 N=5 根 K 棒）
- 選項 B：依策略的平均持倉時間決定 N
- 選項 C：多個 N 都算，取最高 IC 的 N 作為該因子的「最佳預測窗口」

> 範例答案：選項 C 最有資訊量，但計算量較大。起步可先用選項 B（N = 策略平均持倉時間），之後有餘裕再擴展為選項 C。

**Q3：市場狀態的分類標準？**

- 選項 A：用 ADX 閾值切（ADX > 25 = 趨勢，< 25 = 盤整）
- 選項 B：用 BB Width 分位數切
- 選項 C：用 ML 分群（無監督，讓資料自己定義狀態）

> 範例答案：先用選項 A 建立可跑的系統，選項 C 是長期目標。選項 A 的好處是因子 IC 表可讀性高、人可以直接理解。

## 三、痛點知識庫（Pain Point Knowledge Base）

### 概念

有了因子庫之後，痛點的本質是：**某個環境下某個因子缺失或權重錯誤**。

```
痛點（舊描述）：Bollinger Reversion 在趨勢市場頻繁被洗出
↓
痛點（因子語言）：缺少「趨勢強度」因子作為環境過濾，
                  在趨勢強度 IC 高的環境下仍在進場
```

### 痛點診斷流程

```
回測結果差
    ↓
1. 識別主要失敗環境（例：ADX > 25 時虧損集中）
    ↓
2. 查因子庫：策略現有哪些因子？哪個在此環境 IC 最低？
    ↓
3. 查因子庫：同語義中有沒有在此環境 IC 更高的替代因子？
    ↓
4. 找到解法 → 更新策略組合
    ↓
5. 將「此環境 + 此因子問題 → 此解法」記錄到 pain-point-routing.md
```

### pain-point-routing.md 格式

```markdown
| 痛點類型           | 失敗環境       | 缺失因子     | 推薦替代因子/模組     | 來源策略       |
|------------------|--------------|------------|-------------------|--------------|
| 趨勢市場逆勢被洗出  | ADX > 25     | 趨勢強度過濾  | ADX 正規化因子      | Turtle Trading |
| 假突破太多         | BB Width 低  | 波動率狀態   | BB Width 收縮偵測   | VCP           |
| OOS 大幅衰退       | 任何          | —          | Walk-Forward 換訓練窗口 | ML 通用規則  |
```

### 知識庫與因子庫的關係

```
因子庫   = 底層資料結構（IC 評分表）
痛點路由 = 因子庫的診斷介面（快取層）
```

痛點路由表本質上是「已診斷過的因子缺口 + 解法」的快取：
- 第一次遇到某類問題 → 查因子庫診斷，找解法
- 之後再遇到 → 直接查路由表，跳過診斷步驟

### 待決定的設計問題

**Q4：痛點路由表要自動更新還是人工更新？**

- 選項 A：每次回測後，人工判斷是否有新痛點並更新
- 選項 B：自動從回測報告的 Pain Points 欄位提取，觸發更新流程
- 選項 C：選項 B 自動提取 + 人工確認後才寫入

> 範例答案：選項 C。自動提取保證不遺漏，人工確認保證品質。完全自動化容易產生低品質的路由規則。

**Q5：路由表要怎麼處理「解法有多個」的情況？**

- 選項 A：只保留最高 IC 的一個
- 選項 B：保留前 N 個，加上適用條件描述
- 選項 C：加上「解法評分」欄位，讓使用者依情境選擇

> 範例答案：選項 B（保留前 3 個 + 適用條件）。解法通常有環境依存性，單一最佳解法在不同策略裡效果不同。

## 四、工作循環

每一輪策略研究的標準流程：

```
BEFORE（研究開始前）
  1. 查 pain-point-routing.md → 此策略類型的已知風險
  2. 查 factor-library.md → 有哪些現成因子可以直接用
  3. 執行 backtest-preflight skill

DURING（研究過程中）
  4. 遇到痛點 → 先查路由表，再決定是自己解還是借鑑因子庫
  5. 臨時建立的解法 → 記錄在 experiments/，標記「候選因子」

AFTER（研究結束後）
  6. 更新 capability-profile.md（分數、新痛點、已解決的痛點）
  7. 新因子升級為正式因子 → 加入 factor-library.md，補充 IC 評分
  8. 新路由規則 → 更新 pain-point-routing.md
  9. 跨策略教訓 → knowledge-base/knowledge/backtest-records/
```

## 五、關於 RAG 的判斷

**現階段不需要 RAG**，理由：

- 策略數 < 100 個時，index.md 可直接放進 CLAUDE.md context
- 痛點查詢是結構化的（維度、環境、分數），Grep 比向量搜尋更精確
- RAG 的價值在非結構化全文搜尋，不在此場景

**考慮引入 RAG 的條件**：
- 策略數 > 100，或
- 實驗紀錄累積超過幾百篇，index 文件超過 context 上限
